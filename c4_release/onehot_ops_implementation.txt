IMPLEMENTATION DETAILS
==============================================================================

Lookup Table Pattern:
---------------------
One-hot operations use a **lookup table** pattern where each (i, j) input
pair maps to a specific output value via a dedicated unit.

Key Insight: ONE UNIT PER INPUT PAIR
- Each operation creates one cancel-pair per (i, j) input combination
- Each unit fires ONLY when the specific pair (i, j) is active
- The unit's output is routed to the correct result position k

MUL Operation (emit_mul):
--------------------------
Creates 2 * base * base = 512 units (for base=16)
- One cancel-pair for each of 256 possible (i, j) pairs
- Each pair routes to output position k = (i * j) % base

For each (i, j) pair:
  Unit 2*(i*base+j):     (positive unit)
    W_up[unit, a_reg + i] = S
    W_gate[unit, b_reg + j] = 1.0
    W_down[out_reg + k, unit] = 1/S
    where k = (i * j) % base

  Unit 2*(i*base+j)+1:   (negative unit, for stability)
    W_up[unit, a_reg + i] = -S
    W_gate[unit, b_reg + j] = -1.0
    W_down[out_reg + k, unit] = 1/S

How it works:
- When input is (a[i]=1, b[j]=1), only the unit for pair (i,j) fires
- up[unit] = S * a[i] = S (when a[i]=1)
- gate[unit] = 1.0 * b[j] = 1.0 (when b[j]=1)
- hidden[unit] = silu(S) * 1.0 ≈ S
- output_delta[k] = S / S = 1.0 (at position k = i*j % base)
- Result: one-hot vector with 1 at position k

DIV Operation (emit_div):
--------------------------
Creates 2 * base * (base-1) = 480 units (for base=16, excluding div by 0)
- One cancel-pair for each of 240 possible (a, b) pairs (b ≠ 0)
- Each pair routes to output position q = a // b

Pattern is identical to MUL, except:
- Iterates over a ∈ [0, 15], b ∈ [1, 15] (excludes b=0)
- Output position: q = a // b

MOD Operation (emit_mod):
--------------------------
Creates 2 * base * (base-1) = 480 units (for base=16, excluding mod by 0)
- One cancel-pair for each of 240 possible (a, b) pairs (b ≠ 0)
- Each pair routes to output position r = a % b

Pattern is identical to MUL/DIV, except:
- Iterates over a ∈ [0, 15], b ∈ [1, 15] (excludes b=0)
- Output position: r = a % b

==============================================================================
KEY TECHNIQUES
==============================================================================

1. **SwiGLU Multiplication for Pair Detection**:
   - W_up detects a[i]=1: produces activation S
   - W_gate detects b[j]=1: produces gate value 1.0
   - hidden = silu(S) * 1.0 = S ONLY when BOTH are true
   - This implements logical AND for one-hot inputs

2. **Per-Pair Units**:
   - Each (i, j) pair gets its own unit
   - Unit fires only for that specific pair
   - No spurious activations from other pairs
   - Critical for correctness of lookup table

3. **Routing via W_down**:
   - Each unit routes to the appropriate output position
   - Multiple units may route to the same output (e.g., (2,6) and (3,4) both route to k=12)
   - W_down entries aggregate contributions from all relevant pairs

4. **Cancel Pair for Stability**:
   - Positive unit: +S activation, +1.0 gate
   - Negative unit: -S activation, -1.0 gate
   - Both contribute +1/S to output via W_down
   - Cancels numerical errors for robust computation

==============================================================================
DESIGN EVOLUTION
==============================================================================

Initial Attempt (FAILED):
--------------------------
Tried to create 2*base units (32 for base=16), one cancel-pair per output value k.
For each k, set weights for ALL pairs (i, j) where (i*j) % base = k.

Problem:
- W_up[unit_k, i] was set for multiple i values (all i where ∃j such that i*j % base = k)
- For k=0: ALL values of i and j had weights set (since 0*j=0 and i*0=0 for all i,j)
- When input was (3, 4), unit 0 would fire even though 3*4 ≠ 0
- Cause: W_up[0, 3] set from pair (3, 0), W_up[0, 20] set from pair (4, 4)
- Result: Multiple spurious activations, incorrect outputs

Fix:
----
Create one cancel-pair per (i, j) input pair, not per output value.
- Each unit corresponds to exactly one input pair
- Unit fires ONLY when that specific pair is active
- No spurious activations
- Cost: More units (512 vs 32 for MUL), but correctness is guaranteed

==============================================================================
COMPUTATIONAL COST
==============================================================================

Unit Count:
-----------
- MUL: 2 * 16 * 16 = 512 units
- DIV: 2 * 16 * 15 = 480 units (excludes division by zero)
- MOD: 2 * 16 * 15 = 480 units (excludes modulo by zero)

Weight Sparsity:
----------------
- W_up: 1 non-zero entry per unit (at position a_reg + i)
- W_gate: 1 non-zero entry per unit (at position b_reg + j)
- W_down: Multiple entries per output position (for all pairs mapping to that output)
- Overall: Highly sparse (<<1% of weights are non-zero)

Example for MUL:
- Total weights: 512 * (dim_in + dim_out) ≈ 512 * (160 + 16) ≈ 90K parameters
- Non-zero: 512 * 3 = 1,536 entries
- Sparsity: 98.3% sparse

Register Layout:
----------------
- Input a: positions 0-15 (one-hot vector)
- Input b: positions 16-31 (one-hot vector)
- Output: positions 32-47 (one-hot vector)
- Total: 48 dimensions used

USAGE EXAMPLE
==============================================================================

In a computation graph, you can now express nibble arithmetic:

  # Multiply two nibbles
  product_node = MUL(nib_a, nib_b)  # Returns (a * b) % 16

  # Compute quotient and remainder
  quotient_node = DIV(dividend, divisor)  # Returns dividend // divisor
  remainder_node = MOD(dividend, divisor)  # Returns dividend % divisor

  # Example: Division with remainder for 13 / 3
  quotient = DIV(13, 3)   # Returns 4
  remainder = MOD(13, 3)  # Returns 1
  # Verify: 13 = 4 * 3 + 1 ✓

These operations enable:
- Arithmetic on encoded values
- Decomposition of 32-bit operations into nibble operations
- Implementation of algorithms requiring multiplication/division
- Modular arithmetic for cryptography/hashing

# Autoregressive I/O - State Machine Design

**Goal:** Design a state machine for multi-step I/O generation in transformer weights

---

## State Machine Overview

### PUTCHAR Flow (4 States)

```
State 0 (Normal): VM execution
  ↓ (PUTCHAR detected)
State 1 (Emit THINKING_END): Generate </thinking> token
  ↓ (THINKING_END emitted)
State 2 (Emit Byte): Generate output byte from AX
  ↓ (Byte emitted)
State 3 (Emit THINKING_START): Generate <thinking> token
  ↓ (THINKING_START emitted)
State 4 (Emit VM Step): Generate normal VM step
  ↓ (STEP_END emitted)
State 0 (Normal): Resume normal execution
```

### PRTF Flow (More Complex)

```
State 0 (Normal): VM execution
  ↓ (PRTF detected)
State 1 (Emit THINKING_END): Generate </thinking> token
  ↓ (THINKING_END emitted)
State 2 (Format String Parse): Extract format string from memory
  ↓ (Parse complete)
State 3 (Emit Output): Generate formatted output bytes
  ↓ (For each byte in output)
State 3 (Loop): Continue until output complete
  ↓ (All bytes emitted)
State 4 (Emit THINKING_START): Generate <thinking> token
  ↓ (THINKING_START emitted)
State 5 (Emit VM Step): Generate normal VM step
  ↓ (STEP_END emitted)
State 0 (Normal): Resume normal execution
```

## State Tracking in Transformer

### Approach 1: Lookback-Based State Detection

**Mechanism:** Model looks back at recent tokens to determine state

```python
# Dimension: IO_STATE_0 (not in I/O)
# Always 1.0 unless in I/O mode

# Dimension: IO_STATE_1 (just emitted THINKING_END)
# Set to 1.0 when last token was THINKING_END

# Dimension: IO_STATE_2 (just emitted byte)
# Set to 1.0 when last token was a byte (0-255) AND IO_STATE_1 was active

# Dimension: IO_STATE_3 (just emitted THINKING_START)
# Set to 1.0 when last token was THINKING_START AND IO_STATE_2 was active
```

**Implementation via Attention:**
- L14 attention: Look back 1-3 tokens
- Query: CONST (always query)
- Key: Match last token types (THINKING_END, BYTE, THINKING_START)
- Value: Copy state flags
- Output: Set IO_STATE_* dimensions

**Challenge:** Attention typically looks at markers, not token values directly

### Approach 2: Explicit State Dimensions (Recommended)

**Mechanism:** Maintain state dimensions that persist across steps

```python
# Add to _SetDim:
IO_STATE = 326  # 326-329 (4 dims: state machine for I/O generation)
  # IO_STATE[0]: Normal execution
  # IO_STATE[1]: Emit THINKING_END next
  # IO_STATE[2]: Emit output byte(s) next
  # IO_STATE[3]: Emit THINKING_START next

# State transitions in L15 FFN:
# When PUTCHAR detected → set IO_STATE[1] = 1
# When IO_STATE[1] + generated THINKING_END → set IO_STATE[2] = 1
# When IO_STATE[2] + generated byte → set IO_STATE[3] = 1
# When IO_STATE[3] + generated THINKING_START → set IO_STATE[0] = 1
```

**Implementation:**

1. **L5 FFN:** Detect PUTCHAR → set IO_PUTCHAR_PENDING flag
2. **L6 Attention:** Carry forward IO_STATE dimensions
3. **L13 FFN:** Update IO_STATE based on transitions
4. **L15 FFN:** Use IO_STATE to control output routing

**Problem:** How does model know what token it just generated?

### Approach 3: Multi-Head Output (Best for Transformer)

**Mechanism:** Generate multiple tokens in parallel, select based on state

```python
# L15 generates multiple "candidate" outputs:
# - Candidate 0: THINKING_END (if PUTCHAR detected)
# - Candidate 1: AX byte (if just emitted THINKING_END)
# - Candidate 2: THINKING_START (if just emitted byte)
# - Candidate 3: Normal VM step (if just emitted THINKING_START)

# Selection head (L16 if we had it, or in L15 FFN):
# - If PUTCHAR_DETECTED → select candidate 0 (THINKING_END)
# - If LAST_WAS_THINKING_END → select candidate 1 (byte)
# - If LAST_WAS_BYTE → select candidate 2 (THINKING_START)
# - Else → select candidate 3 (normal)
```

**Problem:** Standard transformers generate 1 token at a time, not candidates

## Practical Solution: Runner-Assisted State Machine

**Hybrid Approach:** Transformer generates markers, runner orchestrates

### Modified Generation Loop

```python
class AutoregressiveVMRunner:
    def __init__(self, ..., conversational_io=False):
        self.conversational_io = conversational_io
        self.io_state = 'NORMAL'  # State machine in runner

    def run(self, bytecode, data, ...):
        context = self._build_context(bytecode, data, ...)
        output = []

        while True:
            # Generate next token
            logits = self.model.forward(context)
            next_token = torch.argmax(logits[-1]).item()

            # State machine
            if self.conversational_io:
                next_token = self._io_state_machine(next_token, context)

            context.append(next_token)

            # Check for output or halt
            if next_token == Token.HALT:
                break

        return output

    def _io_state_machine(self, next_token, context):
        """Manage I/O state transitions."""

        # Detect I/O opcode in context
        if self._is_io_opcode(context):
            if self.io_state == 'NORMAL':
                # Intercept: generate THINKING_END
                self.io_state = 'EMIT_BYTE'
                return Token.THINKING_END

        # After THINKING_END, emit output byte
        if self.io_state == 'EMIT_BYTE':
            byte_val = self._extract_output_byte(context)
            self.io_state = 'EMIT_THINKING_START'
            return byte_val

        # After byte, emit THINKING_START
        if self.io_state == 'EMIT_THINKING_START':
            self.io_state = 'EMIT_VM_STEP'
            return Token.THINKING_START

        # After THINKING_START, resume normal generation
        if self.io_state == 'EMIT_VM_STEP':
            self.io_state = 'NORMAL'
            # Let model generate VM step normally

        return next_token  # No interception
```

**Pros:**
- Manageable complexity
- State machine in Python (easier to debug)
- Model still generates all content (bytes come from model's AX)
- Truly autoregressive (model's AX determines output)

**Cons:**
- Runner intercepts token generation
- Not "pure" transformer (has external state machine)
- Requires modified generation loop

**Is this autoregressive enough?**
- ✅ Model generates VM state (PC, AX, SP, BP)
- ✅ Output bytes extracted from model's AX register
- ⚠️ Runner controls when to emit thinking tags and bytes
- ⚠️ Not "pure" transformer (runner has state machine)

## Pure Transformer Solution: Embedding-Based State

**Insight:** Use embeddings to track state within the model

### Implementation

**Step 1: Add State Tokens**

```python
# New tokens for state tracking
Token.IO_EMIT_BYTE = 274  # Internal marker: "emit byte next"
Token.IO_EMIT_THINKING_START = 275  # Internal marker: "emit thinking start next"
VOCAB_SIZE = 276
```

**Step 2: Modified Generation Sequence**

```
Normal flow:
  REG_PC ... STEP_END

PUTCHAR flow:
  REG_PC ... (PUTCHAR detected)
  THINKING_END           ← Model generates this
  IO_EMIT_BYTE          ← Model generates this (internal marker)
  <byte>                ← Model sees IO_EMIT_BYTE, generates byte
  IO_EMIT_THINKING_START ← Model generates this (internal marker)
  THINKING_START        ← Model sees IO_EMIT_THINKING_START, generates this
  REG_PC ... STEP_END   ← Resume normal
```

**Step 3: L15 FFN State-Based Routing**

```python
# Detect last token = IO_EMIT_BYTE → output byte from AX
if last_token_is(IO_EMIT_BYTE):
    generate_byte_from_AX()

# Detect last token = IO_EMIT_THINKING_START → output THINKING_START
if last_token_is(IO_EMIT_THINKING_START):
    generate(THINKING_START)

# Detect PUTCHAR opcode → start sequence
if PUTCHAR_detected:
    generate(THINKING_END)
    generate(IO_EMIT_BYTE)  # Sets up next state
```

**Pros:**
- Pure transformer (state in embeddings)
- No runner state machine
- Truly autoregressive

**Cons:**
- More complex weight engineering
- Internal markers visible in token stream
- Model must learn to recognize its own markers

## Recommendation: Embedding-Based State (Pure Transformer)

**Use internal state tokens** for pure autoregressive I/O:

```
Token.THINKING_START = 272
Token.THINKING_END = 273
Token.IO_STATE_EMIT_BYTE = 274  # Internal: emit byte next
Token.IO_STATE_EMIT_THINKING = 275  # Internal: emit THINKING_START next
VOCAB_SIZE = 276
```

**Generation sequence for PUTCHAR:**
```
Step 1: REG_PC(5) + REG_AX(5) + ... + STEP_END(1)  # Normal
Step 2: THINKING_END(1)  # Exit thinking
Step 3: IO_STATE_EMIT_BYTE(1)  # Internal marker
Step 4: <byte>(1)  # Output from AX
Step 5: IO_STATE_EMIT_THINKING(1)  # Internal marker
Step 6: THINKING_START(1)  # Resume thinking
Step 7: REG_PC(5) + REG_AX(5) + ... + STEP_END(1)  # Next step
```

**Total tokens per PUTCHAR:** 35 (normal) + 5 (I/O sequence) = 40 tokens

**Implementation in weights:**
- L5: Detect PUTCHAR → set flag
- L13: State routing logic
- L15: Generate sequence based on last token

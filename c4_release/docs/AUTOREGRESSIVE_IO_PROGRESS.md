## Architecture Overview

### Token Flow (Conversational Mode)

**Before (Legacy Mode):**
```
User: Run program
Model: <generates VM state tokens>
Python: output.append("Hello")  # Side effect!
Model: <generates more VM state tokens>
```

**After (Conversational Mode):**
```
User: Run program
Model: <generates thinking>
Model: <generates VM step, detects PRTF>
Model: <generates THINKING_END token>
Model: Hello  # Generated by transformer!
Model: <generates THINKING_START token>
Model: <generates next VM step>
```

### Key Difference
- **Legacy:** I/O is Python side effect
- **Conversational:** I/O is part of token stream (100% autoregressive)

## Benefits of Autoregressive I/O

1. **100% Transformer-Based:** No Python code for I/O
2. **Visible Output:** Users see output in conversation naturally
3. **Interactive:** Can handle user input from messages
4. **Testable:** Can verify transformer generates correct output
5. **Compliant:** Meets testing checklist requirement #4

---

## Technical Challenges

### 1. Weight Design Complexity
- Need to detect I/O opcodes from PC
- Generate thinking tags at right time
- Handle variable-length output
- Track state across thinking tag boundaries

### 2. Format String Parsing
- Printf needs to extract format string from memory
- Extract arguments from stack
- Format output correctly
- All must happen in transformer weights

### 3. Context Management
- Output appears mid-generation
- Need to track position in user input
- Attention must handle thinking tags
- Context window constraints

## Root Causes

### 1. Architecture Regression (n_heads 8→16)
Commit ee35744 changed `n_heads` from 8→16, reducing head dimension (HD) from 64→32. This broke attention score budgets throughout the model, causing massive OUTPUT values (~106k).

**Fix**: Reverted `n_heads` back to 8 at line 731.

### 2. Residual Activation Problem
FFN units were firing spuriously due to residual values in dimensions like:
- `ALU_LO/HI`, `AX_CARRY_LO/HI` at PC/AX byte positions
- `OUTPUT_BYTE_LO/HI` at marker positions
- `FETCH_LO/HI` at non-IMM operations

Units designed to fire at markers (MARK_PC=1, IS_BYTE=0) were also firing at byte positions (MARK_PC=0, IS_BYTE=1) due to residuals.

**Solution**: Add blockers for position flags (`IS_BYTE`, `MARK_PC`) and opcodes (`OP_JMP`, `OP_EXIT`).

### 3. Unprogrammed Unit Crossfire
Random weights in unprogrammed L6 FFN units caused spurious activation when OUTPUT_BYTE dimensions had residual values from I/O operations.

**Solution**: Zero out units with OUTPUT_BYTE weights writing to OUTPUT dimensions.

### Fix 4: Part 8c/8g - Crossfire Patch for L6 FFN

**File**: `neural_vm/vm_step.py`
**Lines**: 1765-1784 (in `set_vm_weights`, after ALL L6 FFN setup)

**Problem**: Unprogrammed L6 FFN unit 1128 was firing on OUTPUT_BYTE_LO residuals at PC marker for EXIT, writing +40 to OUTPUT_LO[0], causing prediction of 0 instead of 10.

**Detection Logic**:
1. Check if unit writes to OUTPUT_LO or OUTPUT_HI (W_down weights > 0.01)
2. Check if unit has strong OUTPUT_BYTE weights in W_up (> 50)
3. If both: zero out entire unit (W_up, W_gate, W_down, biases)

**Code**:
```python
# === BUG FIX 2026-04-09 (part 8c/8g): Patch spurious L6 FFN units ===
# Unprogrammed FFN units with random weights can fire spuriously when:
# - OUTPUT_BYTE dimensions have residual values (from IO operations)
# - Unit writes to OUTPUT_LO/HI dimensions
# This causes incorrect output at marker positions (especially PC marker).
# Example: unit 1128 fires on OUTPUT_BYTE_LO residuals at PC marker for EXIT,
# writing to OUTPUT_LO[0] and causing prediction of 0 instead of 10.
patched_count = 0
for u in range(4096):
    # Check if writes to OUTPUT
    writes_output_lo = ffn6.W_down[BD.OUTPUT_LO:BD.OUTPUT_LO+16, u].abs().max().item() > 0.01
    writes_output_hi = ffn6.W_down[BD.OUTPUT_HI:BD.OUTPUT_HI+16, u].abs().max().item() > 0.01
    if not (writes_output_lo or writes_output_hi):
        continue

    # Check for strong OUTPUT_BYTE weights
    output_byte_lo_weight = ffn6.W_up[u, BD.OUTPUT_BYTE_LO:BD.OUTPUT_BYTE_LO+16].abs().max().item()
    output_byte_hi_weight = ffn6.W_up[u, BD.OUTPUT_BYTE_HI:BD.OUTPUT_BYTE_HI+16].abs().max().item()

    if output_byte_lo_weight > 50 or output_byte_hi_weight > 50:
        # Zero out this unit
        ffn6.W_up[u, :] = 0
        ffn6.W_gate[u, :] = 0
        ffn6.W_down[:, u] = 0
        ffn6.b_up[u] = 0
        ffn6.b_gate[u] = 0
        patched_count += 1
```

**Critical**: This patch MUST run after ALL L6 FFN setup functions:
- `_set_layer6_routing_ffn` (lines 1640-1756)
- `_set_io_putchar_routing` (creates units 900-1200)
- `_set_io_binary_pop_marker_ffn` (creates more units)

**Impact**: Zeroed 8 spurious units, fixing EXIT 0 test.

## Technical Details

### Signal Strength
- `S = 20` (standard signal strength)

### Position Flags
- `MARK_PC`: 1.0 at PC marker, 0.0 elsewhere
- `MARK_AX`: 1.0 at AX marker, 0.0 elsewhere
- `IS_BYTE`: 1.0 at byte positions, 0.0 at marker positions
- `MARK_SP`, `MARK_BP`: Similar markers for stack/base pointers

### Dimension Ranges
- `OUTPUT_LO`: 174-189 (16 nibbles)
- `OUTPUT_HI`: 190-205 (16 nibbles)
- `OUTPUT_BYTE_LO`: 480-495 (overlaps with TEMP)
- `OUTPUT_BYTE_HI`: 496-511 (overlaps with TEMP)
- `ALU_LO/HI`: ALU computation dimensions
- `AX_CARRY_LO/HI`: AX register carry dimensions
- `FETCH_LO/HI`: Instruction fetch dimensions

### Blocker Strategy
- **Weak blocker**: `-S` (suppresses by 20)
- **Medium blocker**: `-S * 8` to `-S * 10` (suppresses by 160-200)
- **Strong blocker**: `-S * 20` (suppresses by 400)

Blocker strength chosen based on:
1. Expected residual magnitude
2. How critical it is to prevent firing
3. Existing activation pattern (bias + other weights)

## Lessons Learned

1. **Architecture changes have cascading effects**: The n_heads change broke attention score budgets globally, affecting all operations.

2. **Residual values are pervasive**: Dimensions designed for specific positions (markers) retain residual values at other positions (bytes), causing spurious activation.

3. **Blocker placement is critical**: Position-based blockers (IS_BYTE, MARK_PC) are essential to prevent crossfire between markers and byte positions.

4. **Unprogrammed units need patching**: Random initialization in unused FFN units can cause spurious activation on residuals.

5. **Patch timing matters**: The crossfire patch must run AFTER all unit creation, not during.

6. **Multiple layers can contribute to same bug**: JMP bug involved both L6 (OP_JMP blocker) and L10 (IS_BYTE blockers).

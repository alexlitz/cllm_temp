# Autoregressive I/O - Architecture Analysis

**Date:** 2026-04-07
**Analysis:** Understanding the complexity of truly autoregressive I/O

## Current TOOL_CALL Mechanism (Reference)

### How Tool Calling Works

**Step 1: L5 FFN - Opcode Detection**
```python
# Detects I/O opcodes at AX marker
# OPEN=30, READ=31, CLOS=32, PRTF=33, GETCHAR=64, PUTCHAR=65
for opcode in [OPEN, READ, CLOS, PRTF, GETCHAR, PUTCHAR]:
    # 2-way AND: OPCODE_BYTE_LO[lo] AND OPCODE_BYTE_HI[hi] AND MARK_AX
    # → sets IO_IS_TOOL_CALL ≈ 5.0 when active
```

**Step 2: L6 Attention Head 5 - Relay to SE**
```python
# Relays IO_IS_TOOL_CALL from AX marker → SE position
# Q: NEXT_SE (query at SE position)
# K: MARK_AX (attend to AX marker)
# V: copy IO_IS_TOOL_CALL
# O: write to CMP[2]
```

**Step 3: L6 FFN - Generate TOOL_CALL Token**
```python
# When CMP[2] AND NEXT_SE both active:
# - Set NEXT_TOOL_CALL → emit TOOL_CALL token instead of STEP_END
# - Clear NEXT_SE → suppress STEP_END
```

**Result:** Single token swap (STEP_END → TOOL_CALL)

## Proposed Conversational I/O Mechanism

### What We Need

**For Printf:**
```
Normal flow:
  <thinking>VM step, PRTF detected</thinking>STEP_END
  <thinking>VM step 2</thinking>STEP_END
  ...

Conversational flow:
  <thinking>VM step, PRTF detected</thinking>THINKING_END
  Hello World\n  ← Generated by transformer!
  THINKING_START
  <thinking>VM step 2</thinking>STEP_END
  ...
```

**Challenge:** Variable-length output, not just a single token swap!

### Architectural Challenges

#### Challenge 1: Variable-Length Generation

- Tool calling: Swaps 1 token (STEP_END → TOOL_CALL)
- Conversational I/O: Generates N tokens (THINKING_END + output bytes + THINKING_START)
- Model needs to know when output is complete
- No fixed-length sequence

#### Challenge 2: Format String Parsing

Printf requires:
1. Extract format string pointer from stack
2. Read format string from memory (variable length)
3. Extract arguments from stack (variable count)
4. Parse format specifiers (%d, %x, %c, %s)
5. Format each argument
6. Generate output bytes

This is complex logic that's hard to encode in transformer weights!

#### Challenge 3: State Management

The model needs to track:
- Position in format string
- Which argument we're on
- Whether we're inside/outside thinking tags
- Current PC, AX, SP, BP (normal VM state)

#### Challenge 4: Output Length Unknown

- Simple printf("Hi\n") → 3 bytes
- Complex printf("%d + %d = %d\n", a, b, a+b) → variable length
- Model can't know output length in advance
- Needs END_OUTPUT marker or similar

## Architecture Options

### Option A: Hybrid Approach (Recommended)

**Computation:** 100% autoregressive (current, working)
**I/O formatting:** Python side effects (current, working)
**I/O presentation:** Runner wraps output in thinking tags

**Implementation:**
```python
# In runner (legacy mode)
output = []
# ... VM execution with Python I/O handlers ...
return output

# In runner (conversational mode)
output_parts = []
for step in vm_execution():
    if step.has_io():
        output_parts.append("</thinking>\n")
        output_parts.append(step.io_output)  # From Python handler
        output_parts.append("\n<thinking>")
    output_parts.append(step.vm_tokens)  # Normal VM step
return ''.join(output_parts)
```

**Pros:**
- Easy to implement (2-3 hours)
- Uses existing Python I/O handlers
- Output wrapped in thinking tags
- Still "conversational" from user perspective

**Cons:**
- Not truly 100% autoregressive for I/O
- I/O formatting happens in Python
- Doesn't meet strict interpretation of testing requirement

### Option B: Truly Autoregressive (Complex)

**Everything:** 100% autoregressive through transformer

**Implementation Requirements:**
1. **Format String Decoder** in weights
   - Attention mechanism to extract format string
   - State machine for parsing %d, %x, %c, %s
   - 8-12 hours of weight engineering

2. **Argument Formatter** in weights
   - Integer → decimal string conversion
   - Integer → hex string conversion
   - Character formatting
   - String extraction from memory
   - 12-16 hours of weight engineering

3. **Output State Machine** in weights
   - Track position in output
   - Know when to emit THINKING_START/END
   - Handle variable-length sequences
   - 8-12 hours of weight engineering

4. **Runner Integration**
   - Remove Python I/O handlers in conversational mode
   - Trust model's generated output completely
   - 2-3 hours

**Total Effort:** 30-43 hours (4-6 days)

**Pros:**
- Truly 100% autoregressive
- No Python code for I/O
- Meets strict testing requirement

**Cons:**
- Extremely complex weight engineering
- Difficult to debug
- High risk of bugs
- May hit model capacity limits

### Option C: Simplified Autoregressive

**PUTCHAR only:** Generate single bytes autoregressively
**PRTF:** Use Python formatter (fallback)

**Implementation:**
```
PUTCHAR flow:
  <thinking>PUTCHAR detected, AX=65</thinking>THINKING_END
  A  ← Generated from AX
  THINKING_START
  <thinking>Next step</thinking>STEP_END

PRTF flow:
  <thinking>PRTF detected</thinking>STEP_END
  [Python formats output using existing handler]
```

**Effort:** 8-12 hours

**Pros:**
- Proves concept with simple case (PUTCHAR)
- Can extend to PRTF later
- Manageable complexity

**Cons:**
- Only partial autoregressive I/O
- Printf still uses Python

## Technical Specification for Option C

### L5 FFN: PUTCHAR Detection

```python
# Detect PUTCHAR opcode (65 = 0x41)
unit = 450
ffn.W_up[unit, BD.OPCODE_BYTE_LO + 1] = S  # lo nibble = 1
ffn.W_up[unit, BD.OPCODE_BYTE_HI + 4] = S  # hi nibble = 4
ffn.b_up[unit] = -S * 1.5
ffn.W_gate[unit, BD.MARK_AX] = 1.0
ffn.W_down[BD.IO_IS_PUTCHAR, unit] = 10.0 / S
```

### L6 Attention: Relay PUTCHAR Flag

```python
# Head 6: relay IO_IS_PUTCHAR from AX → SE
base = 6 * HD
attn.W_q[base, BD.NEXT_SE] = L
attn.W_q[base, BD.MARK_AX] = -L
attn.W_k[base, BD.MARK_AX] = L
attn.W_v[base + 1, BD.IO_IS_PUTCHAR] = 1.0
attn.W_o[BD.CMP + 3, base + 1] = 1.0  # Store in CMP[3]
```

### L6 FFN: Thinking Tag State Machine

```python
# State machine:
# 1. When PUTCHAR detected → NEXT_THINKING_END
# 2. After byte generated → NEXT_THINKING_START
# 3. Then normal STEP_END

# Need state tracking - this is the hard part!
# May need additional dimension for state tracking
```

**Challenge:** How to track "we just emitted THINKING_END, now emit byte, then emit THINKING_START"?

**Possible Solution:** Multi-step generation with runner state

## Alternative: Runner-Based Conversational Mode

**Simplest approach:**

```python
class AutoregressiveVMRunner:
    def run(self, ..., conversational_io=False):
        if conversational_io:
            return self._run_conversational()
        else:
            return self._run_legacy()

    def _run_conversational(self):
        """Wrap VM execution with thinking tags."""
        parts = []
        parts.append("<thinking>\n")
        parts.append("Running C program...\n")

        for step in self._execute_vm():
            if step.opcode in [Opcode.PRTF, Opcode.PUTCHAR]:
                # Exit thinking for output
                parts.append("</thinking>\n")
                parts.append(step.output)  # From Python handler
                parts.append("\n<thinking>\n")
                parts.append(f"Output generated: {repr(step.output)}\n")
            else:
                # Normal step inside thinking
                parts.append(f"Step {step.num}: PC={step.pc}, AX={step.ax}\n")

        parts.append("</thinking>\n")
        parts.append(f"Program completed. Exit code: {exit_code}\n")
        return ''.join(parts)
```

**Effort:** 3-4 hours
**100% Autoregressive?:** No, but conversational from user perspective
**Testing Compliance:** Debatable
